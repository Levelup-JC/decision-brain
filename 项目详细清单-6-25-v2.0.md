# Decision Brain 项目详细清单

更新时间：2026-06-25

这份清单用于给后续接手的人直接对接。内容按“项目目标 -> 已完成 -> 当前进度 -> 未完成 -> 接口与路径”组织。

## 1. 项目最初的目标

Decision Brain 的目标不是自动交易，而是做一个可被 Agent 装载的投资决策大脑。核心不是盯盘，而是把“资产记忆、研究、估值、计划、建议”串成一个稳定流程。

最初确定的主流程是：

1. 用户说“我想买 X / 我想加仓 X / 我想卖出 X”
2. 系统先查当前仓位和历史记忆
3. 系统自动做研究
4. 系统先找对标项目，再做估值
5. 系统生成 draft 计划
6. 用户确认后进入 active 监控
7. 后续加仓 / 卖出都基于仓位、估值、事件、thesis、价格曲线综合判断

明确不做：

- 自动交易
- 私钥 / 助记词 / 签名 / 托管
- 高频监控
- 多用户权限系统
- 社交化产品

## 2. 我们一开始的预期

### 2.1 产品预期

一开始的预期非常清晰：

- 不是再做一个行情面板
- 不是让用户每次都手工解释历史背景
- 不是让 Agent 用聊天上下文临时记忆仓位
- 而是做一个“长期记忆 + 研究 + 计划 + 建议”的中台

### 2.2 MVP 预期的 5 件事

PRD 里最早确认的 MVP 只做 5 件事：

1. 记住资产、仓位、成本、历史最高持仓和关键决策
2. 在建仓或关注后，自动完成基础调研和重要事件跟踪
3. 先找对标项目，再给估值区间和计划
4. 在加仓或卖出时，结合仓位、估值、事件、价格曲线和组合暴露给最终建议
5. 每天只做一次新闻 / 事件 / 仓位刷新

### 2.3 设计原则

最早就定下来的规则是：

- 先查仓，再研究，再建议
- 先生成 draft，再等用户确认
- 没有足够证据可以说“不确定”
- 价格曲线只能是输入之一，不能单独决定结论
- 事件不是附属模块，而是核心输入

## 3. 目前已经完成了什么

### 3.1 项目骨架已经跑通

本地服务已经具备完整入口：

- HTTP 服务：`npm start`
- MCP 服务：`npm run mcp`
- Dashboard：`http://127.0.0.1:4177/`

这部分已经在 README 和服务契约里写清楚了。

### 3.2 资产记忆层已经完成

现在系统不是只看当前请求，而是会先查资产历史。

已经完成的能力：

- `lookup-portfolio-memory`
- 读取当前持仓
- 读取历史资产
- 读取归档资产
- 读取 trace
- 扫描本地组合文件

关键输出字段包括：

- `hasCurrentPosition`
- `hasHistoricalPosition`
- `isArchived`
- `hasPriorResearch`
- `knownAliases`
- `matchedSources`
- `confidence`
- `suggestedIntentClass`

这意味着用户只要说“我想买 X”，系统会先判断：

- 是首投
- 是加仓
- 是回补
- 是重新关注

### 3.3 候选资产入口已经完成

已经有统一的候选入口：

- `evaluate_candidate`

它会自动生成：

- `decisionPack`
- `investmentMemo`
- `positionPlaybook`
- `decisionLicense`

并且会根据历史画像自动切换 `investmentContextClass`：

- `first_buy`
- `add_to_winner`
- `average_down_review`
- `reentry_after_exit`
- `resume_watch_only`

### 3.4 研究层已经结构化

研究层不是一个长文档，而是结构化输出。

现在已经有：

- 项目 thesis
- catalysts
- risks
- comparables
- listing path draft
- funding / unlock draft
- factual / inferred signals
- source ledger

同时，研究状态还能判断：

- `usable`
- `thin`
- `blocked`

这样系统不会把 mock 研究当成真实研究。

### 3.5 加仓 / 卖出建议层已经完成

建议层已经能综合这些维度：

- 当前仓位占比
- 风险等级
- 估值区间
- price curve state
- 研究状态
- 当前 thesis
- 最近事件状态
- 底仓规则
- 同赛道暴露

这部分已经不是“只看涨跌”的简单规则了。

### 3.6 Bitget 接入骨架已经搭好

Bitget 这一侧已经做了第一层接线：

- 5 个 Skill 映射
- market-data MCP 接入骨架
- 研究刷新入口 `refresh_research`
- 连接态识别
- 来源写回 `source ledger`

目前默认的研究数据源是公开 market-data MCP，默认 URL 是：

`https://datahub.noxiaohao.com/mcp`

当前研究流不依赖交易凭证。

### 3.7 Dashboard 和 Handoff 已完成

已经有：

- 本地 Dashboard
- HTTP API
- MCP API
- 给其他 Agent 用的 handoff 目录
- OpenClaw 对接说明

### 3.8 测试已经通过

最新测试结果：

- `npm test`
- 30 / 30 通过

覆盖的关键行为包括：

- 历史识别
- 首投 / 加仓 / 回补判断
- 研究状态判断
- 加仓 / 卖出建议
- 每日监测节奏

## 4. 当前进展到什么地步

如果按产品阶段判断，当前已经不是概念稿，而是一个能跑起来的中后期 MVP。

### 4.1 已经能跑通的主链路

现在至少可以跑通：

1. 用户提资产
2. 系统查仓位历史
3. 系统判断是首投 / 加仓 / 回补 / 重新关注
4. 系统做研究
5. 系统生成估值和计划
6. 用户确认计划
7. 系统进入监测
8. 用户再来问加仓 / 卖出
9. 系统输出建议

### 4.2 目前状态判断

按当前代码和测试状态看：

- 记忆层：基本完成
- 流程层：基本完成
- 建议层：基本完成
- 研究层：大体完成，但还没完全吃满
- Bitget 研究接入：可用，但还需要更多真实使用验证
- UI 展示：可用，但还可以更清晰

我对整体进度的判断是：

- P0 已经完成大半
- 约 80% - 85% 的主链路已经到位
- 剩下主要是“研究补全”和“展示/接入打磨”

## 5. 目前还没完成的工作

### 5.1 真实研究补全还不够满

当前缺口主要集中在：

- 融资背景
- 解锁节奏
- 筹码 / 抛压
- 上所路径
- 潜在上所
- 更完整的项目基本面
- 更稳的事件持续跟踪

现在系统已经能识别“自己不知道”，但还没有把“缺的信息”全部自动补齐。

### 5.2 structured drafts 的写入路径还可以增强

目前已经有：

- `comparablesDraft`
- `listingPathDraft`
- `fundingUnlockDraft`

但后续还可以加更直接的写入 / 更新入口，让人类或 Agent 可以主动补草稿，而不是只靠自动生成。

### 5.3 Dashboard 还能更直观

现在能看，但还可以继续优化成：

- 每个研究块展开显示
- 明确标出哪些是事实、哪些是推断
- 明确标出哪些字段还缺
- 明确标出为什么当前建议被 block

### 5.4 资产识别还可以继续增强

对一些 symbol 不唯一、链上/交易所/股票都可能撞名的资产，还能再做更强的歧义消解。

### 5.5 真实 Bitget 研究流程还需要更多验证

当前已经有接入骨架，但后续还要持续确认：

- 连接态是否稳定
- market-data MCP 返回是否稳定
- 5 个 Skill 的产出质量是否足够
- 研究刷新是否真的能改善决策建议

## 7. 其他人接手时的建议

### 7.1 先接什么

优先顺序建议是：

1. 先接 HTTP / MCP
2. 再看记忆层
3. 再看研究状态
4. 再看建议层
5. 最后才看 Dashboard

### 7.2 先验证什么

最重要的验证顺序：

1. `lookup-portfolio-memory`
2. `evaluate-candidate`
3. `manage-position`
4. `refresh-research`
5. `review-add-intent`
6. `review-sell-intent`
7. `run-daily-monitor`

### 7.3 这版最核心的产品逻辑

这版项目最核心的逻辑其实只有一句话：

**不要让 Agent 每次都从零开始想仓位、研究和计划，而是让它先查记忆，再补研究，再出建议。**
