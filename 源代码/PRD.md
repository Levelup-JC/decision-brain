## 1. 一页确认版

### 1.1 产品一句话

Decision Brain 是一个可被 Agent 装载的投资决策大脑。  
用户只需要一句话表达意图，比如“我建仓了 X”“我想加仓 X”“我想卖出 X”，系统就负责完成仓位记忆、项目调研、对标估值、计划生成、事件跟踪和最终投资建议。

### 1.2 MVP 只做 5 件事

1. 记住你的资产、仓位、成本、历史最高持仓和关键决策。
2. 在你建仓或关注某个项目后，自动完成基础调研和重要事件跟踪。
3. 先找对标项目，再给估值区间和计划，而不是直接拍脑袋给点位。
4. 在你想加仓或卖出时，结合仓位、估值、事件、价格曲线和组合暴露，给出最终投资建议。
5. 每天只做一次新闻更新、事件跟踪和仓位刷新，避免过度打扰。

### 1.3 MVP 明确不做什么

本期不做：

- 自动交易
- 私钥、助记词、签名、托管资产
- 高频监控和高频提醒
- 复杂量化策略执行
- 多用户权限系统
- 社交化产品

### 1.4 产品定位

这个产品不是“自动下单机器人”，而是“投资顾问大脑”。

它的价值不在交易那一端，而在前面的决策层：

- 帮你记住你到底为什么买
- 帮你持续跟踪项目有没有变好或变差
- 帮你把“感觉要卖了”变成“基于证据的卖出建议”
- 帮你的 Agent 真的理解你的仓位和计划，而不是每次都从零开始聊天

### 1.5 计划确认区：这 12 条先拍板

以下 12 条，是开发前必须确认的核心结论：

1. **产品定位**  
   这是“投资决策大脑”，不是自动交易机器人。
2. **交互方式**  
   优先一句话交互，不以表单录入为核心。
3. **启动方式**  
   从“我建仓了一个项目 / 我想关注一个项目”开始，而不是先做行情终端。
4. **决策顺序**  
   必须先调研，再找对标，再出估值，再生成计划，最后才给行动建议。
5. **计划状态**  
   所有计划都要先生成 `draft`，只有用户确认后才变成 `active`。
6. **监控频率**  
   新闻、事件、仓位默认 24 小时最多更新一次。
7. **事件输入来源**  
   重要事件不能依赖用户手填，系统要主动调研和持续跟踪。
8. **建议表达方式**  
   前台统一展示“最终投资建议”，不显示生硬的 `ALLOW / BLOCK / DELAY`。
9. **卖出逻辑**  
   卖出建议必须基于估值区间、事件进展、仓位阶段、底仓规则和 thesis 状态。
10. **加仓逻辑**  
   加仓建议必须结合总组合占比和同赛道暴露，不能只看单个代币。
11. **安全边界**  
   不碰私钥，不自动清仓，不自动执行交易。
12. **Bitget 资源定位**  
   Bitget 提供数据、分析和接入能力；Decision Brain 负责记忆层、计划层和建议层。

### 1.6 MVP 成功标准

如果 MVP 满足下面 8 条，就算成功：

1. 用户能一句话开始管理一个资产。
2. 系统能自动形成结构化调研摘要。
3. 系统能找到对标对象并给出三档估值。
4. 系统能从估值生成 `draft` 计划。
5. 用户确认后，计划能进入 `active`。
6. 用户发起“加仓 / 卖出”时，系统能输出结构化最终投资建议。
7. 系统能每天最多一次更新新闻、事件和仓位状态。
8. 所有关键判断都能留下清晰的 Decision Trace，不会记忆混乱。

## 3. P0 规则总表

这一章不是技术实现，而是产品规则。这里必须确认清楚。

### 3.1 输入规则

1. 支持三类资产：
   - 股票 / ETF
   - 主流加密货币 / 交易所已上市代币
   - 链上小币 / 合约地址资产
2. 用户不需要一次性提供所有信息，系统允许先记住不完整仓位。
3. 如果组合总规模未知，系统照常建档，但所有加仓建议必须提示“组合视角不完整”。
4. 如果用户给的是自然语言计划，也要解析成结构化计划字段。

### 3.2 价格曲线规则

价格曲线可以作为判断输入，但不能单独决定建议。

价格曲线在系统里的用途：

1. 判断当前是否追高。
2. 判断当前处于上涨、震荡、回撤还是加速阶段。
3. 判断当前价格位于估值区间的哪个位置。
4. 判断市场是否已经提前交易某个利好。

结论：

- 可以“基于代币价格走势曲线”做判断。
- 但不能只看曲线，必须叠加估值、事件、仓位和 thesis。

### 3.3 底仓规则

不建议对所有资产统一使用“最高总持仓量的 20%”。

更合理的默认规则是按风险等级分层：

| 资产类型 | 默认底仓建议（按历史最高持仓计算） |
| --- | --- |
| 股票 / ETF | 20%-40% |
| BTC / ETH / SOL 等主流币 | 15%-30% |
| 中市值加密资产 | 10%-20% |
| 链上小币 / meme / 高波动资产 | 0%-10% |

规则结论：

1. 底仓基准优先用“历史最高持仓”，不是当前持仓。
2. 风险越高，默认底仓越低。
3. 用户可覆盖默认值，但系统必须有清晰默认值。

### 3.4 加仓规则

加仓不是“看涨就加”，而是一个组合管理动作。

加仓建议至少要检查 6 件事：

1. 当前单资产占组合多少。
2. 当前同赛道总暴露多少。
3. 当前价格是否接近估值上沿。
4. 当前是否处于明显追高阶段。
5. thesis 是否比建仓时更强。
6. 最近是否出现新的利好或利空。

默认输出不是“能加 / 不能加”，而是：

- 是否建议加仓
- 建议最多加到总组合多少
- 如果不建议加，现在缺的是什么条件

### 3.5 卖出规则

卖出建议必须同时结合 5 类因素：

1. **估值位置**  
   当前是否已经进入保守 / 基准 / 乐观估值区间。
2. **事件进展**  
   已兑现什么，未兑现什么，还有没有后续催化剂。
3. **仓位阶段**  
   当前是否已经接近底仓线，卖完后还剩多少。
4. **thesis 状态**  
   原始买入逻辑是否仍成立。
5. **价格状态**  
   当前是高位放量、事件冲顶，还是普通波动。

默认卖出建议应该优先输出：

- 建议卖多少
- 为什么现在卖
- 为什么不建议一次性卖太多
- 卖完后还剩多少底仓

### 3.6 止盈规则

MVP 不建议用一个死板公式管理所有资产。  
更好的方案是“三层止盈”：

1. **估值止盈**  
   进入基准估值区间，开始分批减仓。
2. **倍数止盈**  
   例如 2x 先回本金，3x 或更高再继续减仓。
3. **事件止盈**  
   重要利好兑现后，如果估值已不便宜，允许提高卖出优先级。

推荐默认模板：

1. 进入基准估值区间：减仓 15%-25%
2. 达到 2x：优先回本金
3. 进入乐观估值区间：继续减仓 20%-40%
4. thesis 未失效时，保留底仓

### 3.7 止损规则

MVP 建议支持三类止损，而不是只看跌了多少：

1. **thesis 止损**  
   项目逻辑失效、产品没进展、团队失信、赛道叙事坍塌。
2. **事件止损**  
   黑客攻击、核心资产异动、重大监管风险、下架、融资失败。
3. **时间止损**  
   60-90 天没有任何正向进展，强制进入复盘。

补充规则：

- 对链上小币，事件止损和流动性止损比固定跌幅更重要。
- 对股票和主流币，可额外参考更严格的价格风险阈值。

### 3.8 事件规则

事件不是附属提醒模块，而是核心决策输入。

MVP 重点跟踪这 8 类事件：

1. 上了哪些交易所
2. 还有哪些潜在交易所没有上
3. 为什么理论上有资源但还没上
4. 融资与背书变化
5. 解锁与代币释放节奏
6. 流动性变化
7. 鲸鱼 / 大户 / 机构动作
8. 重大新闻、合作、监管、产品更新

系统需要区分两类信息：

- **确认事实**
- **推断判断**

如果只是推断，必须明确标记。

### 3.9 监控频率规则

默认节奏：

- 新闻更新：每 24 小时最多 1 次
- 事件跟踪：每 24 小时最多 1 次
- 仓位刷新：每 24 小时最多 1 次

只有重大事件才把计划状态从 `active` 提升到 `needs_review`。

### 3.10 建议展示规则

前台统一用下面这套结构：

```text
最终投资建议：
一句话结论

建议动作：
关注 / 建仓 / 小幅加仓 / 继续持有 / 分批止盈 / 暂缓卖出 / 重新复盘

建议仓位：
不超过总组合 X%
或卖出 X%-Y%

核心理由：
3-5 条

主要风险：
2-3 条

什么会改变这个建议：
具体事件 / 数据变化 / 时间节点

下一次提醒：
事件 / 价格区间 / 时间
```

## 4. MVP 主流程

### 4.1 主流程总览

```text
用户建仓 / 关注一个项目
→ 系统识别资产
→ 系统记录仓位
→ 系统自动调研项目
→ 系统寻找对标项目并生成估值区间
→ 系统生成 draft 计划
→ 用户确认计划
→ 系统开始每日一次监测
→ 用户发起加仓 / 卖出 review
→ 系统输出最终投资建议
→ 系统保存 Decision Trace
```

### 4.2 核心原则

1. 先研究，再建议。
2. 先生成计划，再进入长期跟踪。
3. 没有足够证据时，可以输出“不确定”。
4. 不因为用户情绪化想卖，就直接支持清仓。
5. 不因为价格上涨，就默认建议追高。

## 5. P0 功能详细需求

这一章按“目标 - 触发 - 输入 - 输出 - 关键规则”写，方便开发和验收。

### 5.1 资产识别

**目标**  
把用户口中的资产统一成系统可识别对象，支持股票、主流币和链上小币。

**触发**

- 我建仓了 X
- 我想关注 X
- 我想加仓 X
- 我想卖出 X

**输入**

- ticker / symbol
- 合约地址
- 项目别名
- 自然语言描述

**输出示例**

```json
{
  "assetId": "asset_zora",
  "symbol": "ZORA",
  "name": "Zora",
  "assetType": "onchain_token",
  "chain": "base",
  "contractAddress": "0x...",
  "aliases": ["zora"],
  "riskLevel": "high"
}
```

**关键规则**

1. 股票优先按 ticker 识别。
2. CEX 代币优先按交易所 symbol 识别。
3. 链上小币优先按合约地址识别。
4. 同一个资产只能有一个标准主键。

### 5.2 仓位记忆

**目标**  
系统必须拥有稳定、可更新的仓位记忆，否则后续建议不可信。

**输入**

- 当前持仓数量
- 均价
- 当前价格
- 组合总规模
- 历史最高持仓数量
- 现金 / 稳定币比例

**输出示例**

```json
{
  "assetId": "asset_zora",
  "units": 100000,
  "averageCost": 0.08,
  "currentPrice": 0.11,
  "marketValue": 11000,
  "peakUnits": 120000,
  "portfolioPct": 3.5,
  "sectorExposurePct": 9.2
}
```

**关键规则**

1. 同时记录“当前仓位”和“历史最高仓位”。
2. 底仓规则优先基于“历史最高仓位”。
3. 如果组合总规模未知，后续建议必须提示信息不完整。
4. 仓位刷新默认一天一次。

### 5.3 自动调研

**目标**  
当用户建仓或关注一个项目后，系统主动完成最小可用的投研整理。

**触发**

- 新建仓
- 新关注
- 调研缺失
- 旧调研过期

**调研内容**

1. 项目做什么
2. 赛道与核心叙事
3. 团队与融资背景
4. 投资机构
5. 已上哪些交易所
6. 潜在还可能上哪些交易所
7. 为什么理论上有资源但还没上
8. 流动性如何
9. 是否存在控盘 / 做市 / 出货风险
10. 解锁节奏如何
11. 链上持仓集中度如何
12. 鲸鱼或机构有没有动作
13. 最近有哪些重要新闻
14. 核心 catalyst 是什么
15. 核心风险是什么

**输出示例**

```json
{
  "summary": "项目摘要",
  "thesis": ["核心 thesis 1", "核心 thesis 2"],
  "catalysts": ["待观察事件 1", "待观察事件 2"],
  "risks": ["主要风险 1", "主要风险 2"],
  "exchangeStatus": {
    "listed": ["bitget"],
    "potential": ["binance", "coinbase"]
  }
}
```

**关键规则**

1. 调研输出要结构化，不写成长报告。
2. 关键结论要尽量附来源。
3. 结论和来源分开存储。
4. 重大事件出现后，允许调研增量更新。

### 5.4 对标估值

**目标**  
先找到对标对象，再建立预期估值，替代粗糙静态卖出公式。

**输入**

- 项目调研摘要
- 当前价格 / 市值 / FDV
- 可对标项目列表
- 市场环境和流动性信息

**输出示例**

```json
{
  "comparables": [
    {
      "symbol": "PEER1",
      "type": "direct",
      "reason": "同赛道、同阶段、相近 token 模型"
    }
  ],
  "scenarios": [
    {
      "name": "conservative",
      "fdvRange": [300000000, 600000000],
      "priceRange": [0.12, 0.2],
      "conditions": ["thesis 只部分成立"],
      "actionHint": "谨慎持有或小幅补仓"
    },
    {
      "name": "base",
      "fdvRange": [700000000, 1200000000],
      "priceRange": [0.22, 0.38],
      "conditions": ["核心 catalyst 正常兑现"],
      "actionHint": "优先考虑回本金或分批止盈"
    },
    {
      "name": "bull",
      "fdvRange": [1500000000, 2500000000],
      "priceRange": [0.45, 0.8],
      "conditions": ["出现顶级交易所上所或强采用"],
      "actionHint": "进入高位分批退出区"
    }
  ]
}
```

**关键规则**

1. 必须分保守、基准、乐观三档。
2. 每档都要写成立条件、失败条件、建议动作。
3. 对标对象至少区分直接对标、龙头对标、上行对标。
4. 没有足够好的对标时，可以明确写“不确定”。

### 5.5 投资计划生成

**目标**  
把调研和估值转成用户可确认、可执行、可复盘的计划。

**输入**

- 仓位数据
- 估值模型
- 用户偏好
- 默认底仓规则

**输出示例**

```json
{
  "status": "draft",
  "floorRule": {
    "basis": "peak_units",
    "ratio": 0.1
  },
  "addPlan": "低于保守估值下沿且 thesis 未失效时，可小幅补仓",
  "holdPlan": "保守到基准估值之间继续观察",
  "sellPlan": "进入基准估值区间开始分批卖出",
  "principalRecoveryRule": "达到 2x 或进入基准估值时优先回本金",
  "reviewTriggers": [
    "重大上所消息",
    "解锁前 7 天",
    "流动性下降 30%",
    "90 天 thesis 无进展"
  ]
}
```

**关键规则**

1. 初始状态只能是 `draft`。
2. 用户确认前，不进入正式监控。
3. 卖多少、补多少、留多少底仓都必须写进计划。
4. 计划必须带上“为什么这样设计”。

### 5.6 计划确认

**目标**  
让系统长期管理某个资产前，先得到用户确认。

**输入**

- draft 计划
- 用户确认动作

**输出**

- 计划状态从 `draft` 变为 `active`
- 记录确认时间
- 启动每日监控资格

**关键规则**

1. 未确认的计划不能进入正式监控。
2. 用户后续可重新确认新版本计划。
3. 一旦关键事件改变 thesis，状态可升级为 `needs_review`。

### 5.7 每日跟踪

**目标**  
系统代替用户做低频但有效的持续跟踪。

**跟踪范围**

- 价格与估值位置变化
- 当前仓位刷新
- 上所 / 下架
- 融资
- 合作
- 解锁
- 流动性变化
- 链上大户动作
- 新闻和情绪变化

**输出**

- 当日摘要
- 是否触发 `needs_review`
- 是否建议复盘

**关键规则**

1. 默认 24 小时最多执行一次。
2. 无重要变化时，输出“无关键变化”即可。
3. 重大事件发生时，把计划状态改成 `needs_review`。

### 5.8 加仓评估

**目标**  
当用户说“我想加仓 X”时，系统给出基于整体组合的建议。

**输入**

- 当前单资产仓位
- 组合占比
- 同赛道暴露
- 现金 / 稳定币比例
- 当前估值区间
- 事件进展
- 当前价格曲线状态

**输出示例**

```text
最终投资建议：可以小幅加仓，但不建议超过总组合 1.5%。

原因：
1. 当前该资产已占组合 4.8%，同赛道暴露偏高。
2. 价格已进入基准估值区间上沿，不适合大额追高。
3. thesis 未增强到足以支持重仓追加。

下一步：
如果价格回落到保守估值区间，或出现更强确定性利好，再考虑继续加仓。
```

**关键规则**

1. 加仓建议必须结合组合，而不是只看单资产。
2. 输出要包含“建议最大加仓幅度”。
3. 信息不全时，必须指出缺什么。

### 5.9 卖出评估

**目标**  
当用户说“我想卖出 X”时，系统要先讨论逻辑，再给建议。

**输入**

- 用户计划卖出比例
- 当前仓位
- 卖出后剩余仓位
- 当前估值区间
- 是否触发计划卖点
- 已兑现和未兑现的 catalyst
- thesis 是否失效
- 当前价格曲线状态

**输出示例**

```text
最终投资建议：不建议一次性卖出 80%，建议先减仓 20%-30%。

原因：
1. 当前已进入基准估值区间，部分止盈合理。
2. 原始 thesis 尚未失效，仍有未兑现利好。
3. 一次卖出 80% 会跌破历史最高持仓的底仓线。
4. 当前市场情绪偏恐慌，不适合情绪化清仓。

下一步：
等待下一次关键事件确认后再评估是否继续减仓。
```

**关键规则**

1. 卖出建议必须检查是否违反底仓。
2. 要区分“计划内止盈”和“情绪化卖出”。
3. thesis 未失效时，默认反对无理由一把清仓。

### 5.10 决策留痕

**目标**  
让每次关键建议都能回看，形成长期价值。

**需要记录**

- 用户原始意图
- 当时仓位
- 当时价格和估值区间
- 调研摘要
- 事件判断
- 最终投资建议
- 用户是否采纳
- 后续结果

**关键规则**

1. 每条 trace 必须可回溯到当时所用结论和来源。
2. trace 不是普通日志，而是未来复盘和优化的核心资产。

### 5.11 资产归档

**目标**  
当一个资产清仓或不再关注时，结束跟踪但保留历史记录。

**关键规则**

1. 归档后不再主动监控。
2. 历史调研、计划和 traces 必须保留。
3. 后续重新关注时，可恢复上下文而不是重新建档。

## 6. 记忆系统设计

这一章是 MVP 的基础，因为你的核心要求就是“不能记忆混乱”。

### 6.1 记忆设计原则

1. 调研、来源、估值、计划、事件、建议必须分层存储。
2. 不能把所有内容塞进一段长对话记忆里。
3. 每个资产必须有唯一主键。
4. 每次建议都要写 trace，而不是只返回给用户。
5. 监控状态单独存，避免重复提醒。

### 6.2 推荐数据模型

当前 MVP 保持这 9 类结构即可：

- `assets`
- `positions`
- `sources`
- `researchReports`
- `valuationModels`
- `plans`
- `events`
- `traces`
- `monitorState`

### 6.3 为什么这套记忆不会乱

1. 调研和来源分开，后续能知道“结论来自哪里”。
2. 计划和建议分开，后续能区分“原计划”和“临时 review”。
3. 事件和监控状态分开，后续能避免重复提醒。
4. 归档后仍保留历史，不会把清仓资产混在活跃资产里。

## 7. 技术方案

### 7.1 推荐技术栈

| 层 | 技术栈 | 说明 |
| --- | --- | --- |
| Agent 接入层 | MCP Server | 方便接 Codex / Claude / Cursor / 龙虾 |
| 后端 | Node.js + 原生 ESM | 轻量、够快、适合 MVP |
| 数据存储 | 本地 JSON 状态文件 | 本地优先，方便先跑通 |
| 后续存储升级 | SQLite | 等 MVP 稳定后再上 |
| HTTP 接口 | 本地 JSON API | 给龙虾或其他 Agent 调用 |
| 监控任务 | Node script + scheduler | 每日一次即可 |
| Bitget 能力接入 | Adapter 层 | 先接 Skill / MCP，后续再接更深 API |
| 调研接入 | Surf / 外部 research adapter | 负责找事件、新闻、链上和项目信息 |

### 7.2 现阶段为什么先不用更重的架构

当前阶段最重要的不是炫技，而是跑通闭环：

1. 建仓
2. 调研
3. 估值
4. 计划
5. 监控
6. 最终建议
7. 记忆沉淀

在这个阶段，不值得优先投入的东西：

- 复杂前端
- 高频引擎
- 自动交易
- 分布式系统
- 大而全数据库设计

### 7.3 产品形态建议

最合适的形态不是独立 App，而是一个“可被 Agent 装载的功能脑”：

1. **核心产品形态**：本地服务 + MCP 工具
2. **对外展示形态**：一个极简 dashboard + 一套清晰 API / MCP 工具
3. **最佳使用场景**：被龙虾调用，成为你的投资顾问能力层

这个形态最符合你当前目标：

- 好接入
- 好演示
- 好上传 GitHub
- 好讲 Bitget 黑客松故事

## 8. 与 Bitget 黑客松资源的结合方式

### 8.1 Bitget 提供什么

Bitget 已经提供了三层能力：

1. 交易 API
2. 5 个分析 Skill
3. MCP Server 接入能力

### 8.2 Decision Brain 负责什么

Decision Brain 最适合负责三层：

1. **记忆层**：记住仓位、计划、事件、建议、来源
2. **计划层**：把调研和估值变成可确认计划
3. **建议层**：在加仓 / 卖出时给最终投资建议

### 8.3 5 个 Skill 在产品里的作用

| Skill | 在产品里的角色 |
| --- | --- |
| `macro-analyst` | 判断大盘环境、风险偏好、宏观约束 |
| `market-intel` | 看链上、机构、ETF、鲸鱼、流动性 |
| `news-briefing` | 看新闻、上所、融资、合作、监管 |
| `sentiment-analyst` | 看恐慌、FOMO、多空拥挤、情绪极端 |
| `technical-analysis` | 看趋势、支撑阻力、追高风险、价格曲线状态 |

### 8.4 我们不应该重复造什么

不需要重复做：

- 行情基础能力
- 技术指标能力
- 新闻聚合能力
- 链上基础情报能力

我们真正该做的是把这些能力组织成完整决策流程：

- 形成记忆
- 形成计划
- 形成建议
- 形成复盘链

## 9. Demo 路径

黑客松演示建议只跑一个资产，重点展示完整闭环。

### 9.1 Demo 脚本


## 11. 项目执行前需要提前预估和解决的问题

### 11.1 数据可信度问题

问题：

- 链上小币信息容易不完整
- 上所预期经常缺乏公开硬证据
- 流动性、控盘、做市很多时候只能半结构化判断

解决建议：

1. 明确区分“确认事实”和“推断判断”
2. 尽量给出来源
3. 允许输出“不确定，需要进一步调研”

### 11.2 建议过度自信问题

问题：

- 如果系统给出非常绝对的买卖口吻，会显得危险

解决建议：

1. 统一用“最终投资建议 + 理由 + 风险 + 变化条件”的格式
2. 不承诺收益
3. 明确这是投资讨论与仓位管理助手

### 11.3 记忆变乱问题

问题：

- 调研、事件、来源、计划、建议如果混在一起，后面会越来越难维护

解决建议：

1. 坚持 normalized state
2. 来源单独存 `sources`
3. 调研、事件、估值、计划、trace 分层存储

### 11.4 监控过频问题

问题：

- 提醒太多会让用户烦，也会让 Agent 显得没有判断力

解决建议：

1. 默认一天一次
2. 只有重大事件才升级到 `needs_review`
3. 加仓 / 卖出建议以“用户主动发起 review”为主

### 11.5 自动交易诱惑问题

问题：

- 黑客松很容易把故事讲偏成自动交易，但这不是 MVP 当前最强价值

解决建议：

1. 坚持“决策层”定位
2. 交易只作为最末端能力展示，甚至可以不演示
3. 强调这是让 Agent 具备金融记忆和建议能力

## 12. 安全声明

Decision Brain 是投资研究、仓位计划、事件跟踪和决策复盘工具。  
MVP 不自动交易，不保存私钥，不托管资产，不承诺收益。  
所有输出都属于投资讨论与风险管理建议，最终决策由用户自己负责。
