## 1. 现状盘点（已读码确认）

### 1.1 架构是什么样

```
源代码/
├── src/
│   ├── index.mjs              # HTTP 服务入口 (端口 4177)
│   ├── server.mjs             # 路由表：1 个页面 + 13 个 API
│   ├── mcp-server.mjs         # MCP stdio 入口（给龙虾/Claude 挂）
│   ├── service-contract.mjs   # MCP 工具清单（与 HTTP 共用 service 层）
│   ├── data-store.mjs         # 文件型状态存储 data/state.json
│   ├── services/              # 12 个 service（记忆/研究/估值/计划/建议/监测）
│   ├── adapters/
│   │   ├── bitget-adapter.mjs # ★ 5 个 Bitget Skill → market-data MCP 映射
│   │   ├── market-data-parse.mjs
│   │   ├── http-mcp-client.mjs
│   │   └── surf-adapter.mjs   # 仍是 mock
│   └── ui/
│       ├── dashboard.html     # ★ 只读页面（衬线字体、米黄背景）
│       └── dashboard.js       # ★ boot() 只 fetch 一次 /api/state 后渲染
└── test/                      # 7 个测试文件
```

关键事实：
- 后端纯 Node，无任何依赖（package.json 零 dependencies），`npm start` 即起。
- HTTP server 同源托管页面 + API（`server.mjs:39-47` 服务 HTML/JS，`:49-128` 服务 13 个 endpoint）。**前端和 API 同源 = 没有 CORS 问题，这是我们最大的便利，必须守住。**
- 状态全在 `data/state.json`，按 assetId 归一管理 9 个分区（assets/positions/researchReports/sources/valuationModels/plans/events/traces/monitorState）。
- `/api/state`（`api-service.mjs:67` → `stateSummary`）返回 counts + assets + positions + plans + valuationModels + recentEvents + recentTraces。**前端看资产变化的数据源就是它，已经够用。**

### 1.2 现有 13 个 HTTP API（前端可直接调，全部同源）

| # | Method | Path | 作用 | 关键入参 |
|---|--------|------|------|---------|
| 1 | GET | `/api/health` | 健康检查 | - |
| 2 | GET | `/api/capabilities` | 工具契约+agent playbook | - |
| 3 | GET | `/api/state` | 全量状态摘要（前端主数据源） | - |
| 4 | GET | `/api/asset-context?asset=SYM` | 单资产完整上下文 | asset |
| 5 | POST | `/api/lookup-portfolio-memory` | 查仓位历史，判首投/加仓/回补 | assetQuery |
| 6 | POST | `/api/evaluate-candidate` | 生成 Decision Pack/Memo/估值 | assetQuery |
| 7 | POST | `/api/manage-position` | 写入实际持仓+生成 draft 计划 | assetQuery, units, averageCost, currentPrice, portfolioValue, naturalLanguagePlan |
| 8 | POST | `/api/refresh-research` | 调 5 个 Bitget Skill 刷新研究 | assetQuery |
| 9 | POST | `/api/confirm-plan` | draft → active | assetQuery 或 planId |
| 10 | POST | `/api/review-add-intent` | 加仓建议 | assetQuery, portfolioValue |
| 11 | POST | `/api/review-sell-intent` | 卖出建议 | assetQuery, requestedSellPct |
| 12 | POST | `/api/run-daily-monitor` | 每日监测（24h 节流） | force? |
| 13 | POST | `/api/log-source` | 追加来源记录 | assetQuery, title, keyClaim |
| 14 | POST | `/api/archive-asset` | 归档资产 | assetQuery |

> 注意：第 14 个 archive 也在，共 13 个 POST/GET 业务端点。前端**不需要新增任何业务 API**，全部已具备。我们只新增 1 个对话编排端点（见 §3）。

### 1.3 Bitget 接入现状（已用得不错，但要在前端"显形"）

`bitget-adapter.mjs` 已把 **5 个 Bitget 感知 Skill** 映射到公开 market-data MCP（`https://datahub.noxiaohao.com/mcp`，无需 API Key）：

| Bitget Skill | MCP 工具 | 在决策中的角色 |
|--------------|----------|----------------|
| macro-analyst | macro_indicators, rates_yields, cross_asset | 宏观环境 |
| market-intel | defi_analytics, network_status, crypto_market | 市场情报 |

## 2. 目标架构（单进程 / 单源 / 零 CORS）

### 2.1 一句话架构

> 在现有 Node server 上，把只读 dashboard 升级成"聊天壳 + 结构化卡片 + 实时资产看板"三栏 SPA；新增 1 个 `/api/chat` 对话编排端点，内部把自然语言→意图→调已有的 13 个 API；对话层可降级为按钮表单，LLM 不可用时 Demo 照常跑通。

### 2.2 分层图

```
┌─────────────────────────────────────────────────────────┐
│  浏览器 (同源 http://<host>:4177/)                         │
│  ┌──────────────┬──────────────────┬───────────────────┐ │
│  │ 左：聊天对话   │ 中：结构化卡片流   │ 右：实时资产看板    │ │
│  │ (输入框+消息)  │ (Memo/估值/建议)   │ (资产/仓位/事件/   │ │
│  │              │                  │  Bitget Skill 面板) │ │
│  └──────┬───────┴────────┬─────────┴─────────┬─────────┘ │
└─────────┼────────────────┼───────────────────┼───────────┘
          │ POST /api/chat  │ POST /api/* (13)  │ GET /api/state (poll 3s)
          ▼                 ▼                   ▼
┌─────────────────────────────────────────────────────────┐
│  Node HTTP server (server.mjs) —— 同一个进程              │
│  ┌─────────────────────────────────────────────────────┐ │
│  │ 新增：chat-orchestrator.mjs                          │ │
│  │  1. 自然语言 → 意图分类 (LLM 或 关键词规则降级)         │ │
│  │  2. 抽取槽位 (assetQuery/units/sellPct...)            │ │
│  │  3. 调已有 service 层函数 (不重复实现业务)              │ │
│  │  4. 组织结构化回复 + 自然语言总结                       │ │
│  └─────────────────────────────────────────────────────┘ │
│  现有 12 个 service（原样复用，零改动）                     │
│  现有 bitget-adapter（原样复用）                            │
└─────────────────────────────────────────────────────────┘
          │
          ▼ data/state.json（持久化）
```

### 2.3 为什么这样设计（对抗约束的逻辑）

- **为什么不另起 React 工程**：1-2 天 1-2 人，另起工程要配 CORS、配两套部署、调跨域 cookie，风险全在工程化而非功能。增强现有页面 = 所有功能性时间都花在刀刃上。这也是你选的"增强现有原生页面"。
- **为什么对话端点放进同一个 Node 进程**：保持单源，前端 `fetch('/api/chat')` 和 `fetch('/api/state')` 同源，零 CORS。
- **为什么对话层必须可降级**：你选了"真·对话式 Agent（需接 LLM）"，但同时只剩 1-2 天。LLM 编排有不确定性（解析错意图、超时、key 失效）。所以架构上**意图层抽象成接口**，有两个实现：`llmIntent()` 和 `ruleIntent()`。Demo 前一晚如果 LLM 不稳，一行开关切回规则版，按钮照样跑完 evaluate→manage→confirm→review→sell 全链路。**这是不错过提交的保险。**

### 2.4 LLM 选型（已查 ~/.config/financial）

可用凭证（按优先级）：
1. **qiniu / DeepSeek-V4**（OpenAI 兼容，`base_url: https://api.qnaigc.com`）— 默认选这个，OpenAI 格式最省事。
2. **Qwen / 通义千问**（黑客松主办方阿里赞助发的 Token）— **强烈建议至少接一次并在 Demo/说明里点名**，赞助商是 Qwen，用它有印象分。
3. **Anthropic**（`api.anthropic.com`）— 备选。

> 决策：对话编排用 **OpenAI 兼容协议**写（qiniu 和 Qwen 都兼容），通过环境变量切 `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL`。这样换供应商不改代码。**密钥只从 `~/.config/financial/secrets.yaml` 或环境变量读，绝不硬编码进仓库。**

## 3. 新增的唯一后端端点：`POST /api/chat`

这是整个前端交互的核心。其余全部复用现有 API。

### 3.1 契约（前后端对接的硬约定）

请求：
```json
{
  "message": "我想买 BTW，研究一下值不值得",
  "sessionId": "demo-001",
  "context": { "lastAsset": "BTW" }
}
```

响应（必须稳定，前端按此渲染）：
```json
{
  "ok": true,
  "intent": "evaluate_candidate",
  "assetQuery": "BTW",
  "slots": { "units": null, "sellPct": null },
  "actionsTaken": [
    { "api": "lookup-portfolio-memory", "ok": true },
    { "api": "evaluate-candidate", "ok": true }
  ],
  "data": { "...": "对应 API 的原始返回，前端用来渲染卡片" },
  "reply": "BTW 在 BSC 上，当前市值 $X，研究状态待补强。我已生成 Decision Pack...",
  "suggestions": ["生成持仓计划", "刷新 Bitget 研究", "看加仓建议"],
  "degraded": false
}
```

- `intent` 枚举（与现有 API 一一对应）：`lookup_memory | evaluate_candidate | manage_position | refresh_research | confirm_plan | review_add | review_sell | run_monitor | log_source | archive | get_context | smalltalk | unknown`
- `degraded: true` 表示走了规则降级（LLM 没用上），前端可显示一个灰色小标"规则模式"。
- `suggestions` 给前端渲染快捷按钮，点了等于发一条对应 message —— 这就是"聊天 + 结构化卡片"的混合体验落地方式。

### 3.2 编排内部流程（chat-orchestrator.mjs 伪逻辑）

```
1. classifyIntent(message)
   - 主：调 LLM（OpenAI 兼容），system prompt 给出意图枚举 + 槽位 schema，要求返回 JSON
   - 降级：关键词规则（"买/研究"→evaluate；"我有/持仓/成本"→manage；"加仓"→review_add；
           "卖/减"→review_sell；"刷新研究"→refresh；"确认计划"→confirm）
2. extractSlots(message, intent)  // assetQuery、units、averageCost、sellPct…
3. dispatch(intent, slots) → 调对应的 api-service 函数（直接 import，不走 HTTP 自调）
4. summarize(data) // LLM 把结构化结果转一句人话；降级时用模板拼
5. 返回 §3.1 结构
```

> 关键约束给开发：**编排层禁止重新实现任何业务逻辑**，只能 import 调用 `api-service.mjs` 里已有的导出函数（evaluateCandidate / managePosition / …）。它只做"翻译"，不做"决策"。决策永远在 service 层，保证和 MCP 入口行为一致。

## 4. 前端三栏交互设计（在 dashboard.html/js 内实现）

技术栈：原生 ES module + 一个 CDN 图表库（Chart.js）。不引入构建工具。

### 4.1 布局

```

│ (30%)      │ (40%)               │ (30%)            │
│            │                     │                  │
│ 消息气泡    │ Decision Pack 卡片   │ 资产列表(可点)    │
│ ...        │ Investment Memo 卡片 │ 仓位 + 估值区间   │
│            │ 估值区间条形图        │ 资产价值变化曲线   │

│ │输入框   │ │ (含 coreReasons/     │ Bitget Skill 面板 │
```

### 4.2 必须实现的交互功能点（按优先级）

P0（不做就没 Demo）：
- F1 对话输入框 → 调 `/api/chat` → 渲染消息气泡 + 触发对应卡片
- F2 结构化卡片渲染：Decision Pack / Investment Memo / 估值区间 / 加仓建议 / 卖出建议
- F3 资产看板每 3 秒 poll `/api/state`，资产/仓位/计划数字**实时变化**（这就是"看到资产变化"）
- F4 快捷建议按钮（来自 chat 的 suggestions），点击=发消息，零门槛跑完整链路
- F5 "记录交易"表单：units/averageCost/currentPrice/portfolioValue → 调 `manage-position`（这就是"前端记录交易内容"）

P1（让它酷炫）：
- F6 估值区间条形图（Chart.js）：当前 FDV 在保守/基准/高估值区的位置，红黄绿分区
- F7 资产价值变化曲线：用 traces/state 时间序列画 portfolio value 折线
- F8 Bitget Skill 活动面板：refresh-research 后展示 5 个 Skill 的调用状态/来源数/关键claim（**让评委看见 Bitget 在工作**）
- F9 消息流式打字机效果 + 卡片淡入动画（酷炫感）
- F10 研究就绪度可视化：ready/partial/blocked 用色块；"待补充"字段明确标灰（呼应诚实原则）

P2（有时间再说）：
- F11 深色/科技感主题切换（现在是米黄衬线，偏文艺，可加一个 Agent 控制台风格深色皮肤）
- F12 资产详情抽屉：点资产滑出 `/api/asset-context` 全量上下文

### 4.3 状态变化怎么"看得见"（你的核心诉求）

- 每个写操作（chat 或表单）成功后，前端**立即** re-fetch `/api/state` 刷新右栏，不等 3 秒轮询。
- 资产卡片用数字滚动动画（CountUp）体现 units/价值变化。
- 新增 trace 时右栏 Decision Trace 顶部高亮插入一条，给"系统在思考"的实时感。

## 7. 部署方案（你选了"公开 URL"）

因为是单 Node 进程、零依赖、文件存储，最省事的三选一：

| 方案 | 适合 | 注意 |
|------|------|------|
| **Railway / Render** | 推荐。直接跑 `npm start`，给公网域名 | 文件存储 state.json 重启会丢——Demo 够用；要持久可挂 volume |
| **ngrok** | 最快。本地起服务 `ngrok http 4177` | 临时域名，录视频/评审期间机器要开着 |
| **Vercel** | 不推荐。Vercel 是 serverless，文件型 state + 长连接轮询不契合，要改造，1-2 天不划算 |

> 决策：**Railway**（要稳定可访问地址）或 **ngrok**（要最快出链接）。两者都不需要改代码，只需 env 配 LLM key。
> 部署前 checklist：① 设 `DECISION_BRAIN_DATA_DIR` 到可写目录 ② 设 LLM env ③ `/api/health` 返回 ok ④ 首页能开。

## 8. 红线与诚实原则（评审基线，必须守）

1. **不假造数据**：MCP 给不了的字段（融资轮次、解锁表、thesis、对标估值）一律前端标"待补充"，不编。这恰好是评审"真实可运行"的加分项。
2. **不碰交易**：不接 Bitget 58 个交易 Tools，不存私钥。Demo 里主动说明"我们是决策大脑不是执行器"。
3. **密钥安全**：LLM key 只从 env / `~/.config/financial` 读，不进 git。提交前 `git grep` 查一遍。
4. **后端零改动原则**：service 层和 adapter 不动（除非发现 bug），只新增 chat 编排 + 前端。降低回归风险。
5. **单源零 CORS**：前端所有请求同源 `/api/*`，不引入跨域。
