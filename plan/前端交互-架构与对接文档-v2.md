## 2. v2 目标架构：AI 投资委员会 / 多 Agent 作战室

### 2.1 一句话

> Chief Agent（首席）接收用户意图，**fan-out** 给若干并行的专家 Agent（5 个 Bitget 分析师 + 记忆 Agent + 估值 Agent），各 Agent 独立并发工作、各自把意见"冒泡"到 UI，Chief 收齐后**综合**成最终建议。前端真并发、各 Agent 卡片独立异步出现 = Harness 多 Agent 协作观感。

### 2.2 Agent 角色表（前端要显形的"委员会成员"）

| Agent | 底层实现（已存在，不新写业务） | 输出 |
|-------|-------------------------------|------|
| 🧠 Chief（首席决策官） | chat-orchestrator：意图分类+fan-out+综合 | 最终建议 + 调度日志 |
| 💾 Memory（记忆官） | `lookup-portfolio-memory` | 首投/加仓/回补判定 |
| 📊 Macro（宏观） | bitget-adapter macro skill | 宏观信号 |
| 🔗 On-chain（链上情报） | bitget-adapter market-intel | 市值/流动性/链 |
| 😱 Sentiment（情绪） | bitget-adapter sentiment | 恐贪/多空比 |
| 📈 Technical（技术） | bitget-adapter technical | 指标信号 |
| 📰 News（新闻） | bitget-adapter news | 叙事/事件 |
| 💰 Valuation（估值官） | valuation-service / candidate-service | 估值区间+所处分区 |

> 重点：这些 Agent **全是对现有代码的"重命名+显形"**，不是新业务。Macro/Onchain/Sentiment/Technical/News 直接来自 bitget-adapter 已并行的 5 个 skill 结果；Memory/Valuation 来自现有 service。

### 2.3 分层图

```
┌────────────────────────────────────────────────────────────┐
│ 浏览器 (同源 Vercel 域名)                                      │
│ ┌──────────────┬───────────────────┬──────────────────────┐ │
│ │ 左：与 Chief   │ 中：委员会作战室     │ 右：实时资产看板        │ │
│ │ 对话           │ (N 个 Agent 卡片    │ (资产/仓位/估值/事件/   │ │
│ │ 消息气泡+输入   │  异步并发冒泡)       │  Decision Trace)      │ │
│ │ 快捷建议        │ 各 Agent: 思考中→   │ portfolio 价值曲线     │ │
│ │               │  返回意见(淡入)      │                      │ │
│ └──────┬────────┴─────────┬─────────┴──────────┬───────────┘ │
└────────┼──────────────────┼────────────────────┼─────────────┘
         │ POST /api/chat    │ POST /api/agent/:role (并发 N 个)  │ GET /api/state (poll 3s)
         ▼                   ▼ (这就是"多 Agent 并行"的真相)        ▼
┌────────────────────────────────────────────────────────────┐
│ Node 同一进程（Vercel serverless function）                   │
│  chat-orchestrator.mjs：意图→fan-out 计划→综合                │
│  agent-runner.mjs：按 role 调对应 service/adapter（并行安全）  │
│  现有 12 service + bitget-adapter（原样复用）                  │
└────────────────────────────────────────────────────────────┘
         │
         ▼ storage backend（接口）→ 本地:文件 / Vercel:KV
```

### 2.4 "多 Agent 并行"怎么落地（关键设计）

两种实现，二选一（看时间）：

- **方案甲（推荐，真并行观感最强）**：前端拿到 fan-out 计划后，**对每个 Agent 角色并发发独立请求** `POST /api/agent/:role`，谁先返回谁先在中栏冒泡。这是"真的多个 Agent 在跑"的观感，最 Harness。后端 `/api/agent/:role` 是薄封装，按 role 调对应 service/skill。
- **方案乙（省事，单请求内并行）**：前端只发一个 `/api/chat`，后端内部 `Promise.allSettled` 并行所有 Agent，一次性返回数组。前端用错位淡入动画模拟逐个到达。并行是真的（后端并发），但观感稍弱。

> 决策建议：P0 先做方案乙（保底闭环），有时间升级方案甲（前端并发 + 各 Agent 独立 loading 态）。两者前端渲染同一套 Agent 卡片组件，升级成本低。

### 2.5 为什么这样设计

- **零 CORS**：所有请求同源 `/api/*`。
- **后端零业务改动**：Agent 只是对已有 service/skill 的"角色化封装"，决策逻辑不动 → 和 MCP 入口行为一致、回归风险低。
- **对齐赛道**：Trading Agent 赛道明确建议"为 Agent 设计的管理面板、多 Agent 协同（分析师+执行+风控）"。委员会作战室 = 正中靶心，且**创新度(20%)**直接拉满。

## 3. 新增后端：编排 + Agent runner

### 3.1 `POST /api/chat`（Chief 入口）契约

请求：
```json
{ "message": "我想买 BTW，研究一下值不值得", "sessionId": "demo-001", "context": { "lastAsset": "BTW" } }
```
响应：
```json
{
  "ok": true,
  "intent": "evaluate_candidate",
  "assetQuery": "BTW",
  "fanout": ["memory", "onchain", "sentiment", "technical", "valuation"],
  "agentResults": [
    { "role": "onchain", "status": "ok", "headline": "BTW 在 BSC，市值 $X，流动性 $Y", "data": {} },
    { "role": "sentiment", "status": "ok", "headline": "F&G=26 恐惧区", "data": {} }
  ],
  "reply": "委员会意见：身份已确认(BSC)，情绪偏恐惧，估值待补强。我建议先...",
  "suggestions": ["生成持仓计划", "刷新全部 Agent", "看加仓建议"],
  "degraded": false
}
```
- `intent` 枚举同 v1：`lookup_memory|evaluate_candidate|manage_position|refresh_research|confirm_plan|review_add|review_sell|run_monitor|log_source|archive|get_context|smalltalk|unknown`
- `fanout`：Chief 决定本轮派哪些 Agent；前端据此渲染"待命/思考中"的 Agent 卡片。
- `degraded:true`：LLM 没用上，走规则降级。前端显示"规则模式"小标。

### 3.2 `POST /api/agent/:role`（方案甲用，单 Agent 端点）

请求：`{ "assetQuery": "BTW" }`，`:role` ∈ `memory|macro|onchain|sentiment|technical|news|valuation`
响应：`{ "ok": true, "role": "onchain", "status": "ok", "headline": "...", "data": {}, "tookMs": 820 }`

> 约束：`/api/agent/:role` 与 `/api/chat` 内部**都只调 api-service / bitget-adapter 已有导出**，禁止重新实现业务。

### 3.3 编排伪逻辑（chat-orchestrator.mjs）

```
1. classifyIntent(message)  → LLM(OpenAI兼容) 主，关键词规则降级
2. extractSlots → assetQuery/units/averageCost/sellPct
3. planFanout(intent) → 决定派哪些 Agent（evaluate→全员；review_add→memory+valuation+sentiment…）
4. 方案甲：返回 fanout 列表给前端并发；方案乙：Promise.allSettled 跑完再返回
5. synthesize(agentResults) → LLM 综合成一句话；降级用模板
```

### 3.4 LLM 客户端 `llm-client.mjs`

- **OpenAI 兼容协议**（DeepSeek / 你的 API 中转站 / 千问都兼容）。
- env 切换：`LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL`。默认 **DeepSeek 或中转站**；**Demo 用千问跑一次并在项目说明点名**（赞助商印象分）。
- **降级开关 `CHAT_RULE_ONLY=1`**：完全不调 LLM，纯规则。
- 密钥只从 env / `~/.config/financial/secrets.yaml` 读，**绝不进 git**。

## 4. 存储改造（Vercel 的唯一实质成本）

### 4.1 问题

Vercel serverless 文件系统**不持久**：每次冷启动 `data/state.json` 丢失。"记录交易、看资产变化"必须有持久存储。

### 4.2 方案：storage backend 接口 + 双实现

`data-store.mjs` 已是"整状态一个 blob 读写"，所以抽一层 backend 即可，**12 个 service 零改动**：

```
storage-backend.mjs (接口)
  - readBlob(): Promise<state | null>
  - writeBlob(state): Promise<void>

实现 A：file-backend.mjs    // 本地，包现有 fs 读写（默认）
实现 B：kv-backend.mjs      // Vercel KV (Upstash Redis)，@vercel/kv，单 key 存整 JSON
```

- `DataStore.load/save` 改为调 `backend.readBlob/writeBlob`，`update()` 不变。
- 环境探测：有 `KV_REST_API_URL` 用 KV，否则用文件 → 本地开发和 Vercel 同一套代码。
- 引入唯一依赖 `@vercel/kv`（仅 KV 实现用到）。

### 4.3 并发注意

Vercel 多实例并发写同一 blob 有覆盖风险。Demo 单用户场景可接受；要稳可在 `update()` 加 KV 乐观锁（read-modify-write 带版本号）——P1，时间不够先不做，Demo 说明里标"单用户 Demo 态"。
